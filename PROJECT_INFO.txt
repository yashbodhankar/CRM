10) How the Project Works (End-to-End)
- Frontend (React + Vite) handles UI, route protection, and user state from JWT token.
- Backend (Express) exposes REST APIs under /api/* and enforces auth/permissions per route.
- Database layer uses MongoDB in normal mode.
- Dev offline mode: when DISABLE_AUTH=true, backend can run without Mongo and uses in-memory dev stores.
- Scheduler (node-cron) runs hourly for reminders and lead auto-progression.

Request lifecycle:
1. User logs in from frontend (/login).
2. Backend validates credentials and returns JWT token.
3. Frontend stores token in localStorage and decodes role/name/email for session state.
4. ProtectedRoute checks route permissions/roles before rendering a page.
5. API calls include Bearer token, backend middleware validates token and authorizes action.
6. Controller performs logic (CRUD, conversion, analytics, notifications), then response is returned.

11) User Roles and Access Given
System roles:
- admin
- manager
- sales
- lead
- employee
- customer

Permission model:
- admin: full access (*) to all modules.
- manager: dashboard, analytics, employees (read/write), projects (read/write), tasks (read/write), leads/customers/deals (read/write/delete), notifications read.
- sales: dashboard, leads/customers/deals (read/write/delete), tasks read, notifications read.
- lead: dashboard, analytics, employees (read/write), projects read, tasks (read/write), leads/customers/deals (read/write/delete), notifications read.
- employee: dashboard read, projects read, tasks read and update own, notifications read.
- customer: dashboard read, own project read access, own-project task read access, notifications read.

Main backend access rules by module:
- Auth users management (/api/auth/users, reset-password): admin only.
- Analytics (/api/analytics, /advanced): analytics:read permission.
- Leads (/api/leads): leads:read/write/delete permissions.
- Customers (/api/customers): customers:read/write/delete permissions.
- Deals (/api/deals): deals:read/write/delete permissions.
- Notifications (/api/notifications): notifications:read permission.
- Employees (/api/employees): create admin or manager; update admin/manager/lead; delete admin only.
- Projects (/api/projects):
  - create/update/delete: admin or manager
  - customer update: customer
  - submit completion: lead
  - verify completion: admin or customer
  - milestone/subtask completion updates: admin/manager/lead
- Tasks (/api/tasks):
  - create/delete: admin/manager/lead
  - update: admin/manager/lead/employee
- AI tools (/api/ai): admin/manager/sales/lead.

12) Business Flow of the CRM
Lead-to-Revenue flow:
1. Lead is created (manual or auto-assigned to active sales/lead employee in round-robin).
2. Lead is qualified and tracked in pipeline statuses.
3. Lead scoring endpoint evaluates priority (0-100) with recommendation.
4. Conversion endpoint turns a lead into a deal and marks lead as converted.
5. If customer record does not exist, system creates customer profile.
6. Deal then moves through stages: new -> qualified -> proposal -> negotiation -> converted -> won/lost.

Customer management flow:
1. Customer profile is created with contact/company/status data.
2. System can auto-generate customer login credentials (temporary password) when needed.
3. Team logs interactions (call/email/meeting/note) and uploads customer documents.
4. Timeline endpoint merges interactions + document events in reverse chronological order.

Project and task delivery flow:
1. Managers/admin create projects and assign progress structure.
2. Tasks are created and assigned to team members.
3. Leads/admin/managers update milestone/subtask completion.
4. Lead can submit project completion.
5. Admin/customer verifies completion.
6. Customer can post project-specific updates.

13) Automation and Notifications Flow
- Hourly automation job checks tasks with deadlines in the next 24 hours and sends reminder notifications.
- Hourly automation promotes stale leads (status=new for over 7 days) to qualified.
- Important actions (new lead, lead updates, lead conversion, etc.) trigger in-app notifications.
- Email sending is optional and controlled by SMTP environment variables.
